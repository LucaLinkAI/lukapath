LucaPath Studio A local chat app that runs on your Claude subscription

LucaPath Studio wraps four Chinese destiny-report skills behind a Claude-style chat UI. A React client talks to a thin Express server, which drives the Claude Agent SDK — and the SDK runs entirely on the machine's Claude Code Pro/Max subscription. No Anthropic API key is ever used.

01

What it is

LucaPath Studio is a local, single-user web app. The user chats in Chinese about a person, a family, a child, or business partners; the chat is handled by Claude running one of the project's report skills, which collects birth data turn-by-turn and ultimately writes a polished, self-contained HTML destiny report. That report previews live in a slide-in drawer and can be downloaded or opened in the browser.

Left · Sidebar
New chat, the four report-type shortcuts, and an in-memory "recents" list.
Center · Chat
Streaming conversation that drives the skills. Composer supports file / photo attachments.
Right · Preview Drawer
Generated report HTML, rendered in a sandboxed iframe. Opens automatically on generation.
Core idea

The server is deliberately thin. It does not call the Anthropic REST API. It hands a prompt to the Agent SDK and re-broadcasts the SDK's stream to the browser as SSE. All the intelligence — and all the auth — lives in the local Claude Code runtime.

Four participants — client, server, Agent SDK, and Anthropic — separated by one process/network boundary each. The first three run on your machine; only Anthropic is in the cloud, reached with your subscription's OAuth token (never an API key).

03

Client architecture

A single-page React 18 app built with Vite. All UI state lives in one Zustand store; all styling lives in theme.css driven by CSS variables (a dark navy/gold default and a [data-theme='light'] warm-paper override). The dev server runs on :3000 with strictPort and proxies /api → :8787.

ModuleResponsibility
App.tsxRenders <Landing/> until entered, then <Studio/>.
screens/Landing.tsxAuth gate. Calls GET /api/auth/status; stores AuthStatus (incl. the live model id used by the composer badge).
screens/Studio.tsx2-column grid (Sidebar + Chat). Adds .drawer-open so the chat reflows when the preview drawer slides in.
components/ChatPanel.tsxMessage list + Claude-style composer (attach, drag-drop, paste, model·effort badge). Top bar holds the status dot, drawer toggle, and theme toggle. Generated files render as inline FileCards.
components/ArtifactPanel.tsxFixed slide-in drawer previewing report HTML in a sandboxed iframe. Open-in-browser / refresh / download / close.
components/Sidebar.tsxNew chat, report-type shortcuts, recents.
state/store.tsZustand store: sessions, busy, auth, drawerOpen, theme. Owns send() (upload → stream → attach artifacts).
api/chatStream.tsstreamChat() (parses the SSE wire format), fetchAuthStatus(), uploadFiles().
In-memory by design

There is no client-side persistence beyond the theme preference (localStorage['lp-theme']). Conversations and recents are kept in the Zustand store and reset on reload / server restart.

04

Server architecture

An Express 4 app run directly from TypeScript via tsx. It mounts four route groups, keeps session state in memory, and translates Agent SDK events into SSE frames. The JSON body limit is raised to 32 MB to accept base64 file uploads.

ModuleResponsibility
index.tsBootstraps Express, mounts routes, runs the boot probe (auth + skill discovery), chooses the agent cwd mode.
config.tsPaths (PROJECT_ROOT = repo root holding .claude/skills), EXPECTED_SKILLS, and AGENT_MODEL (pinned to claude-sonnet-4-6; env-overridable).
agent.tsWraps the SDK query(). Builds options, runs a turn, normalizes SDK messages into AgentEvents, captures .html Write paths. Holds probeSkills().
auth.tscheckClaudeAuth(): a tiny probe query reporting { ok, apiKeySource, model }. Cached ~30 s.
sessions.tsIn-memory SessionState map. Each session owns an output/<uuid>/ dir and the SDK session id (for resume).
artifacts.tsdetectArtifacts(): after a turn, finds generated .html (Write events ∪ a non-recursive dir scan); relocates strays into the session dir.
sse.tsMinimal SSE writer: event:/data: frames + a 15 s heartbeat.
routes/chatPOST /api/chat → SSE stream of session / start / token / tool / artifact / error / done.
routes/authGET /api/auth/status.
routes/artifactGET /api/artifact/:sid/:file — whitelisted against session.artifacts (no raw path join → no traversal).
routes/uploadPOST /api/upload — base64 files → output/<uuid>/uploads/, returns absolute paths.
05

How it uses your local Claude Pro/Max subscription

This is the defining trait of the app: no Anthropic API key, no billing through the API. Inference is authenticated and billed exactly as if you were typing into the claude CLI yourself.

The chain of trust

1
You log into Claude Code once terminal
Running claude and signing in with your Claude Pro/Max account stores OAuth credentials under ~/.claude on this machine. The app never sees your password or token — it just relies on this login existing.
2
The server imports the Agent SDK server
@anthropic-ai/claude-agent-sdk's query() launches the Claude Code runtime as a child process, using the claude_code tool preset. That runtime picks up the same ~/.claude credentials.
3
Requests bill against the subscription anthropic
When the runtime calls Anthropic, it presents the subscription's OAuth token — not an API key. The model runs on your Pro/Max quota. The boot probe confirms this: the system/init message reports apiKeySource: "none".
4
The app gates entry on this being healthy client
The landing page calls GET /api/auth/status. If the probe query streams a system/init without an authentication_failed error, you're in. If not, the UI tells you to run claude and log in first.

The exact SDK options

From agent.ts › baseOptions() — the configuration that makes a headless, skill-aware, subscription-backed turn possible:

// server/src/agent.ts
{
  cwd: session.outputDir,            // reports + uploads land here, per session
  model: 'claude-sonnet-4-6',        // pinned via AGENT_MODEL (CLI default is 4.5)
  settingSources: ['user', 'project', 'local'], // loads ../.claude/skills
  permissionMode: 'bypassPermissions',  // headless: no human to approve tool prompts
  allowDangerouslySkipPermissions: true,
  tools: { type: 'preset', preset: 'claude_code' }, // Read/Write/Bash/Skill/…
  includePartialMessages: true,           // token-by-token streaming
  additionalDirectories: [session.outputDir, PROJECT_ROOT],
  resume: session.sdkSessionId             // continue the same conversation each turn
}
  • No API key anywhere. There is no ANTHROPIC_API_KEY in the code or env path; auth is the local subscription via Claude Code.
  • Skills load from the repo. settingSources makes the SDK discover ../.claude/skills; probeSkills() at boot picks the working directory (nested session dir vs. project root) that resolves all four skills.
  • Headless autonomy. bypassPermissions lets the agent use tools (write the report, read uploads) without an interactive approval prompt — there is no human at the CLI.
  • Stateful conversation. The first turn captures session_id from system/init; every later turn passes it as resume, so a skill's multi-step intake flows across messages.
  • Pinned model. AGENT_MODEL forces Sonnet 4.6 so behavior and the in-app badge are explicit (the CLI would otherwise default to Sonnet 4.5).
Implication

Because auth is the machine's login, the app is inherently single-user and local. Whoever is signed into Claude Code is the identity every request runs as. This is the right model for a personal desktop tool, and the wrong model for a multi-tenant hosted service.

06

A chat turn, end to end

1
User sends a message client
store.send() appends the user message and an empty assistant placeholder, flips busy, and (if there are attachments) uploads them first.
2
POST /api/chat opens an SSE stream server
The route resolves/creates the session, emits session + start, then iterates runTurn(). A res 'close' listener aborts the turn if the browser disconnects.
3
Agent SDK runs the turn sdk
query() streams SDK messages. Text deltas → token events; tool_use blocks → tool chips; .html Write paths are captured for artifact detection.
4
Client renders the stream live client
The SSE parser appends tokens to the assistant bubble and shows tool chips (e.g. "技能：lucapath", "写入报告：…html") as they arrive.
5
Artifacts detected after the turn server
detectArtifacts() finds new report HTML and emits an artifact event with a served URL + accent color; then a final done.
6
Drawer opens, file card appears client
The artifact is attached to the assistant message (inline FileCard) and the preview drawer auto-opens to the report.

07

Artifacts & uploads

Outbound — generated reports

Skills are steered to write report HTML into the session's output/<uuid>/ directory (absolute path injected into the prompt). After each turn the server detects them via the union of Write-tool events and a non-recursive directory scan, relocating any stray written outside the session dir back in. They are served read-only through a whitelisted artifact route — the requested file must match a registered artifact, so user input never becomes a raw path.

Inbound — file & photo attachments

The composer accepts files via the + button, drag-and-drop, or paste. On send, they're base64-encoded and POSTed to /api/upload, which saves them into output/<uuid>/uploads/ and returns absolute paths. Those paths are then injected into the prompt so the agent can Read them within the same turn. Uploads live in a subdirectory that the artifact scan ignores, so they're never mistaken for generated reports.

Security posture

Single-user/local, so the threat model is small — but the file routes still avoid path traversal by whitelisting artifacts and sanitizing upload filenames, and report previews run in a sandboxed <iframe>.
